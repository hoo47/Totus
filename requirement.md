# 프로젝트 요구사항 (Project Requirements)

본 문서는 Totus AI Agent Platform 구축을 위한 요구사항과 현재 진행 상황을 정리한 문서입니다.

## 1. 기반 아키텍처 및 대화 관리 (Event-Driven)
- **이벤트 기반 기록**: 각 대화 요소(`user_message`, `assistant_message`, `tool_call`, `tool_result` 등)가 이벤트 단위로 기록되어야 함. (완료)
- **상태 복원 (Fork/Resume)**: 이벤트 기반 아키텍처를 바탕으로 특정 대화(Sequence) 시점부터 대화를 다시 시작하거나 포크(Fork)할 수 있어야 함. (완료)
- **대화 압축 (Compaction)**: 대화가 길어질 경우, 여러 대화를 묶어 LLM을 통해 요약(Compaction)하고 맥락을 유지하면서 토큰 사용량을 최적화해야 함. (완료)

## 2. 에이전트 및 도구 시스템 (Modular Agents & Tools)
- **다중 에이전트 지원**: 단일 에이전트가 아닌, 여러 개의 에이전트를 지원하는 구조여야 함. (완료)
- **도구(Tool) 분리 및 할당**: 에이전트마다 가질 수 있는 도구를 다르게 설정할 수 있어야 함. (완료)
- **글로벌 도구 (Global Tools)**: 모든 에이전트가 런타임에 접근할 수 있거나 공통으로 제공되는 글로벌 도구 기능이 있어야 함. (완료)

## 3. 다중 LLM 프로바이더 통신 (Multi-Provider Support)
- **클라우드 API 통신**: API를 통해 외부 모델(OpenAI, Claude 등)과 직접 통신할 수 있어야 함. (완료)
- **로컬 LLM 통신**: 로컬 환경에 구성된 Ollama 모델과도 통신할 수 있어야 함. (완료)

## 4. 사용자 인터페이스 (Interfaces)
- **CLI 통신**: 터미널 기반의 로컬 환경에서 에이전트와 대화하고 도구를 실행할 수 있어야 함. (완료)
- **웹 대화**: 웹 UI를 통해서도 대화가 가능하여야 함. (진행 대기 - 향후 개발)

## 5. 코어 흐름 (Execution Loop)
- **고정된 파이프라인**: 
  1. 시작 및 초기화
  2. 대화 루프 (메시지 수신 -> LLM 추론)
  3. 도구 실행 (요청 시 권한 검사 및 실행)
  4. 결과 표시 (도구 결과 반환 후 다시 LLM 루프로 전달)
  이라는 확고한 사이클을 지켜야 함. (완료)
